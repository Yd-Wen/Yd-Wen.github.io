# 邀请链长度检查方案归纳

## 问题背景

在邀请裂变系统中，需要确保邀请链不超过最大层级限制（默认 3 级）。当一个已经有下级邀请链的用户绑定到新的邀请人时，可能会形成超过最大层级限制的邀请链。

例如：邀请链 A->B（层级为1）和邀请链 C->D->E（层级为2），当 C 绑定 B 的邀请码时，形成新的邀请链 A->B->C->D->E（层级为4），超过了最大层级限制。

## 方案一：查询直接下级

### 源代码

```javascript
// 检查被邀请人（queryUid）的下级邀请链长度，确保总层级不超过限制
if (queryUid) {
  // 检查被邀请人是否有下级用户
  const inviteesRes = await userCollection.where({
    'inviter_uid.0': queryUid  // 精确匹配数组第一个元素，即直接下级
  }).get()
  
  if (inviteesRes.data.length > 0) {
    // 新的邀请链长度（包括当前用户）
    const newChainLength = inviterUid.length + 1
    
    // 检查是否会超过最大层级限制
    if (newChainLength > MAX_INVITE_LEVEL) {
      throw {
        errCode: ERROR.INVITE_LEVEL_EXCEEDED
      }
    }
  }
}
```

### 代码思想

- 只检查被邀请人的直接下级用户
- 如果被邀请人有直接下级，则假设新的邀请链长度为 `inviterUid.length + 1`
- 检查这个长度是否超过最大层级限制

### 解决的问题/优化方向

- **问题**：只能检查直接下级，无法检查更深层的下级用户
- **优化方向**：需要考虑被邀请人的所有下级用户，而不仅仅是直接下级

## 方案二：递归查询下级

### 源代码

```javascript
/**
 * 递归获取用户的所有下级用户，并计算最大邀请链长度
 * @param {string} uid 用户ID
 * @param {number} currentLevel 当前层级，用于防止无限递归
 * @returns {number} 最大邀请链长度
 */
async function getMaxSubChainLength(uid, currentLevel) {
  // 防止无限递归，超过最大层级限制时停止
  if (currentLevel >= MAX_INVITE_LEVEL) {
    return 0
  }
  
  // 查询当前用户的直接下级
  const subUsersRes = await userCollection.where({
    'inviter_uid.0': uid  // 精确匹配数组第一个元素，即直接下级
  }).field({
    inviter_uid: true
  }).get()
  
  if (subUsersRes.data.length === 0) {
    return 0  // 没有下级用户，返回0
  }
  
  // 递归计算每个下级用户的最大邀请链长度
  let maxChainLength = 0
  for (const subUser of subUsersRes.data) {
    const subChainLength = await getMaxSubChainLength(subUser._id, currentLevel + 1)
    maxChainLength = Math.max(maxChainLength, subChainLength + 1)
  }
  
  return maxChainLength
}

// 使用方法
if (queryUid) {
  // 递归获取被邀请人的所有下级用户，并计算最大邀请链长度
  const maxSubChainLength = await getMaxSubChainLength(queryUid, 0)
  
  // 新的邀请链长度（包括当前用户）
  const newChainLength = inviterUid.length + maxSubChainLength
  
  // 检查是否会超过最大层级限制
  if (newChainLength > MAX_INVITE_LEVEL) {
    throw {
      errCode: ERROR.INVITE_LEVEL_EXCEEDED
    }
  }
}
```

### 代码思想

- 递归查询被邀请人的所有下级用户
- 计算每个下级用户的最大邀请链长度
- 取所有下级用户中的最大邀请链长度
- 检查新的邀请链长度是否超过最大层级限制

### 解决的问题/优化方向

- **解决的问题**：能够检查被邀请人的所有下级用户，包括深层级的下级
- **优化方向**：递归查询可能导致性能问题，特别是当用户数量较多时

## 方案三：降序遍历索引查询

### 源代码

```javascript
/**
 * 通过索引查询方式获取用户的最大下级邀请链长度
 * @param {string} uid 用户ID
 * @returns {number} 最大邀请链长度
 */
async function getMaxSubChainLengthByIndex(uid) {
  // 从 Lmax-2 开始自减到 0，这样只需要查询 Lmax-1 次
  for (let idx = MAX_INVITE_LEVEL - 2; idx >= 0; idx--) {
    // 查询邀请链的指定元素，判断是否包含 queryUid
    const checkRes = await userCollection.where({
      [`inviter_uid.${idx}`]: uid
    }).count()
    
    // 如果查询结果的长度大于0（即有用户的邀请链在该索引位置包含queryUid）
    // 则直接退出查询循环并返回邀请链长度（索引值idx+1）
    if (checkRes.total > 0) {
      return idx + 1
    }
  }
  
  return 0  // 没有找到，返回0
}

// 使用方法
if (queryUid) {
  // 通过索引查询方式获取被邀请人的最大下级邀请链长度
  const maxSubChainLength = await getMaxSubChainLengthByIndex(queryUid)
  
  // 新的邀请链长度（包括当前用户）
  const newChainLength = inviterUid.length + maxSubChainLength
  
  // 检查是否会超过最大层级限制
  if (newChainLength > MAX_INVITE_LEVEL) {
    throw {
      errCode: ERROR.INVITE_LEVEL_EXCEEDED
    }
  }
}
```

### 代码思想

- 利用数据库的索引查询特性
- 从 `MAX_INVITE_LEVEL - 2` 开始降序遍历索引
- 检查每个索引位置是否有用户的邀请链包含 queryUid
- 一旦找到，立即返回对应的邀请链长度

### 解决的问题/优化方向

- **解决的问题**：避免了递归查询的性能问题，查询次数固定
- **优化方向**：查询次数仍然较多（Lmax-1 次），可以进一步优化

## 方案四：特定索引查询

### 源代码

```javascript
// 检查被邀请人（queryUid）的下级邀请链长度，确保总层级不超过限制
if (queryUid) {
  // L1 是 inviterUid.unshift(inviterRecord._id) 前的 inviterUid 的长度
  const L1 = inviterUid.length
  
  // L2 是 inviterUid.unshift(inviterRecord._id) 后的 inviterUid 的长度
  const L2 = L1 + 1
  
  // idx = Lmax - L2
  const idx = MAX_INVITE_LEVEL - L2
  
  // 根据 L2 的长度，queryUid 的 maxSubChainLength 应该小于 Lmax - L2 + 1
  // 否则 newChainLength 就是 Lmax + 1，超出了限制
  // 此时只需查询 queryUid 的邀请链是否有指定索引位置（Lmax - L2 + 1 - 1）等于 queryUid 的用户
  // 如果有用户（长度大于 0），抛出错误
  if (idx >= 0) {
    const checkRes = await userCollection.where({
      [`inviter_uid.${idx}`]: queryUid
    }).count()
    
    if (checkRes.total > 0) {
      throw {
        errCode: ERROR.INVITE_LEVEL_EXCEEDED
      }
    }
  }
}
```

### 代码思想

- 通过数学计算确定需要查询的索引位置
- 只需要查询一次，判断是否有用户的邀请链在该位置包含 queryUid
- 如果有，说明会超出层级限制，直接抛出错误

### 解决的问题/优化方向

- **解决的问题**：查询次数最少，性能最优
- **优化方向**：逻辑更简洁，利用数学计算避免了不必要的查询

## 四种方案比较

| 方案 | 查询次数 | 时间复杂度 | 空间复杂度 | 优点 | 缺点 |
|------|----------|----------|----------|------|------|
| 方案一：查询直接下级 | 1次 | O(1) | O(1) | 简单直接 | 只能检查直接下级，无法检查深层级 |
| 方案二：递归查询下级 | 不固定，取决于用户数量 | O(n) | O(n) | 能检查所有层级 | 性能差，可能导致栈溢出 |
| 方案三：降序遍历索引查询 | Lmax-1次 | O(1) | O(1) | 性能较好，查询次数固定 | 查询次数仍然较多 |
| 方案四：特定索引查询 | 最多1次 | O(1) | O(1) | 性能最优，查询次数最少 | 逻辑稍复杂，需要数学计算 |

## 归纳总结

### 技术原理

1. **邀请链存储结构**：
   - 使用 `inviter_uid` 数组存储邀请链：`[直接邀请人ID, 上级邀请人ID, ...]`
   - 例如：`[B, A]` 表示邀请链 A->B，长度为 2

2. **层级计算**：
   - 邀请链长度 = 数组长度
   - 新的邀请链长度 = 邀请人的邀请链长度 + 被邀请人的最大下级邀请链长度

3. **性能优化**：
   - 从递归查询到索引查询，性能逐步提升
   - 利用数据库索引特性，减少查询次数
   - 通过数学计算，进一步减少查询次数

### 最佳实践

- **方案四** 是最优选择，只需要最多 1 次查询
- 利用数学计算确定需要查询的索引位置
- 避免了不必要的数据库查询，性能最优
- 代码简洁，逻辑清晰

### 应用场景

- 适用于邀请裂变系统中的层级限制检查
- 适用于需要快速判断邀请链长度的场景
- 可以扩展到其他需要层级限制的系统

## 代码优化建议

1. **数据库索引**：
   - 为 `inviter_uid` 字段创建索引，提高查询性能
   - 考虑创建复合索引，优化 `inviter_uid.idx` 查询

2. **错误处理**：
   - 增加错误处理，确保数据库查询失败时能够妥善处理
   - 提供清晰的错误信息，便于调试

3. **参数验证**：
   - 增加参数验证，确保 `queryUid` 等参数的合法性
   - 防止恶意输入导致的安全问题

4. **性能监控**：
   - 添加性能监控，了解查询耗时
   - 优化数据库查询，进一步提升性能

## 结论

通过不断优化，邀请链长度检查的性能从 O(n) 降低到 O(1)，查询次数从多次减少到最多 1 次。方案四通过数学计算和一次查询，完美解决了邀请层级限制的检查问题，是最佳选择。
